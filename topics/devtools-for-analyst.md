## Введение

Browser DevTools (Chrome/Edge/Firefox) - обязательный инструмент системного аналитика для диагностики веб-приложений, анализа API-вызовов и воспроизведения дефектов. DevTools помогает быстро получить факты: какой запрос отправился, что вернулось, где ошибка и как её повторить.

## Основные концепции

### Ключевые вкладки DevTools для аналитика

* **Network:** запросы/ответы, статусы, payload, headers, тайминги.
* **Console:** JS-ошибки, предупреждения, служебные логи.
* **Application/Storage:** cookies, local/session storage, токены.
* **Elements:** структура DOM и быстрый анализ UI-состояний.
* **Performance:** поиск узких мест и долгих операций.

### Что анализировать в первую очередь

* HTTP-статус и тело ответа;
* корреляцию между действиями пользователя и вызовами API;
* наличие ретраев, таймаутов, CORS-ошибок;
* корректность передаваемых параметров и заголовков.

## Практические примеры

### Пример 1: воспроизведение бага "не сохраняется форма"

План:

1. Открыть `Network`, включить `Preserve log`.
2. Повторить сценарий.
3. Найти `POST/PUT` запрос формы.
4. Проверить:
   * request payload;
   * response status/body;
   * серверное сообщение об ошибке.

Результат обычно сразу показывает, где проблема: в валидации клиента, в контракте API или в бизнес-правиле на сервере.

### Пример 2: анализ авторизации

Во вкладках `Network` и `Application` аналитик проверяет:

* наличие access/refresh токенов;
* момент истечения токена;
* реакцию клиента на `401/403`;
* повторную аутентификацию.

Это особенно важно при согласовании нефункциональных требований по безопасности.

### Пример 3: экспорт запроса для разработки и QA

Из `Network` можно скопировать cURL:

```bash
curl 'https://api.example.com/orders' \
  -H 'authorization: Bearer <token>' \
  -H 'content-type: application/json' \
  --data-raw '{"customerId":123,"amount":4500}'
```

Так проще передать воспроизводимый кейс разработчику или тестировщику.

## Рабочие практики

* Используйте `Disable cache` для стабильного воспроизведения.
* Снимайте HAR-файлы для сложных инцидентов.
* Добавляйте в задачу Jira:
  * шаги воспроизведения;
  * cURL/скриншоты;
  * ожидаемое и фактическое поведение.
* Сохраняйте технические выводы в Confluence/DocOps-базе.

## Типичные ошибки и как их избежать

* **Диагностика "на глаз" без Network:** потеря времени и субъективность.  
  **Решение:** фиксировать факты из запросов/ответов.
* **Не учитывается кэш:** баг "плавает".  
  **Решение:** отключать кэш при анализе.
* **Нет артефактов в задаче:** сложно передать контекст.  
  **Решение:** прикладывать HAR, cURL, скриншоты и шаги.

## Связь с другими темами

Тема связана с `HTTP`, `REST API`, `OAuth/JWT`, `Postman`, `Jira и Confluence`, а также с `Мониторингом` при разборе инцидентов.

## Заключение

DevTools для аналитика - это инструмент быстрой технической верификации требований и дефектов. Чем лучше аналитик владеет DevTools, тем быстрее команда приходит к корректному решению.
