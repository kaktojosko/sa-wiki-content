## Введение

**DocOps** - это подход к документации как к продукту и как к процессу, близкому к DevOps. Цель DocOps для системного аналитика: делать документацию актуальной, проверяемой и встроенной в общий жизненный цикл разработки.

В классическом подходе документы часто "живут отдельно" и быстро устаревают. В DocOps документация хранится в репозитории, проходит review, автоматические проверки и публикуется как часть релизного процесса.

## Основные концепции

### Документация как код (Docs as Code)

Базовые принципы:

* хранение документов в Git;
* использование текстовых форматов (`Markdown`, `Asciidoc`, `OpenAPI`, `PlantUML`, `Mermaid`);
* pull request как обязательный этап согласования;
* автоматические проверки качества.

### Структура документации

Для аналитики удобно разделять:

* `docs/business/` - бизнес-контекст, цели, KPI;
* `docs/requirements/` - требования и правила;
* `docs/integrations/` - контракты и взаимодействия;
* `docs/architecture/` - ключевые архитектурные решения;
* `docs/operations/` - эксплуатационные инструкции.

### Автоматизация в DocOps

Типовые автоматические проверки:

* lint markdown (стиль, заголовки, ссылки);
* проверка OpenAPI/JSON Schema;
* валидация диаграмм;
* сборка статического портала документации.

## Практические примеры

### Пример 1: базовый DocOps-пайплайн

```yaml
name: docs-ci
on: [pull_request]
jobs:
  check-docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Markdown lint
        run: npm run lint:md
      - name: OpenAPI validate
        run: npm run lint:openapi
      - name: Build docs
        run: npm run docs:build
```

Результат: аналитик не может слить PR с битой ссылкой или невалидным контрактом.

### Пример 2: workflow изменения требования

1. Создать ветку `feature/SA-310-refund-policy`.
2. Обновить `docs/requirements/refund-policy.md`.
3. Если требование влияет на API, обновить `api/openapi.yaml`.
4. Создать PR и пройти review у разработки/QA.
5. После merge автоматически опубликовать документацию.

### Пример 3: Definition of Done для аналитической документации

Изменение считается завершённым только если:

* обновлены бизнес-правила;
* изменены связанные API/DB-артефакты;
* пройдены автоматические проверки;
* опубликована новая версия документации.

## Типичные ошибки и как их избежать

* **Документация обновляется "потом":** появляются расхождения с системой.  
  **Решение:** обновление документации должно быть частью задачи.
* **Нет единого шаблона:** документы сложно читать и сравнивать.  
  **Решение:** ввести шаблоны для требований, ADR, API-изменений.
* **Ручная публикация:** высокий риск пропустить релевантные изменения.  
  **Решение:** автоматическая публикация из main-ветки.

## Связь с другими темами

DocOps напрямую связан с `Git`, `Swagger/OpenAPI`, `Jira и Confluence`, `LLM для аналитика` и `Мониторингом` (runbook/postmortem-документация).

## Заключение

DocOps превращает документацию из "архива" в рабочий инструмент команды. Для системного аналитика это один из самых эффективных способов повысить качество требований и сократить стоимость изменений.
